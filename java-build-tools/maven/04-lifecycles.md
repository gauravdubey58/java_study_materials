# Chapter 4 — Build Lifecycles

A **lifecycle** is an ordered, named sequence of phases that defines how Maven approaches a specific kind of work. Maven ships with three built-in lifecycles. Understanding them is essential — they are the backbone of every Maven build.

---

## The Three Lifecycles

| Lifecycle | Purpose | Triggered by |
|---|---|---|
| `default` | Compile, test, package, and deploy the project | `mvn compile`, `mvn package`, `mvn deploy`, etc. |
| `clean` | Remove files generated by previous builds | `mvn clean` |
| `site` | Generate project documentation and reports | `mvn site` |

Each lifecycle is completely independent. Running `mvn clean install` runs the `clean` lifecycle and then the `default` lifecycle up to the `install` phase.

---

## Lifecycle 1 — Default

The **default lifecycle** has 23 phases and handles the entire build and deployment process. They always run in order — you cannot skip a phase by jumping ahead.

### Phase Reference Table

| # | Phase | What Happens Here |
|---|---|---|
| 1 | **validate** | Checks that the project POM is correct and all required information is available |
| 2 | **initialize** | Initialises the build state — sets up properties, creates directories |
| 3 | **generate-sources** | Generates source code files (e.g., from WSDL contracts, Avro schemas, Protobuf files) |
| 4 | **process-sources** | Processes generated and regular source code (filtering, annotation processing) |
| 5 | **generate-resources** | Generates resource files to be included in the artifact |
| 6 | **process-resources** | Copies and filters resources from `src/main/resources` to `target/classes` |
| 7 | **compile** | Compiles the project's main Java source code into `target/classes` |
| 8 | **process-classes** | Post-processes the compiled bytecode (e.g., bytecode enhancement for JPA entities) |
| 9 | **generate-test-sources** | Generates test source code |
| 10 | **process-test-sources** | Processes test source code |
| 11 | **generate-test-resources** | Generates resources for tests |
| 12 | **process-test-resources** | Copies and filters test resources to `target/test-classes` |
| 13 | **test-compile** | Compiles the test source code into `target/test-classes` |
| 14 | **process-test-classes** | Post-processes compiled test classes |
| 15 | **test** | Runs unit tests using the configured test framework (JUnit/TestNG). Fails the build if any test fails |
| 16 | **prepare-package** | Performs any last operations needed before packaging (e.g., generates manifests) |
| 17 | **package** | Takes compiled code and packages it into the distributable format: JAR, WAR, or EAR |
| 18 | **pre-integration-test** | Sets up the environment for integration tests (start a server, create a database, etc.) |
| 19 | **integration-test** | Deploys the application and runs integration tests |
| 20 | **post-integration-test** | Tears down the integration test environment (even if tests failed) |
| 21 | **verify** | Runs any checks to verify the package is valid and meets quality criteria |
| 22 | **install** | Installs the artifact into the **local repository** (`~/.m2/repository`) so other local projects can use it |
| 23 | **deploy** | Copies the final artifact to a **remote repository** for sharing with other developers or CI systems |

### Key Phase Relationships

```
validate
  │
  ├── compile           (phase 7) ─── compile sources
  │
  ├── test              (phase 15) ── compile tests → run unit tests
  │
  ├── package           (phase 17) ── create JAR/WAR
  │
  ├── verify            (phase 21) ── integration tests + quality gates
  │
  ├── install           (phase 22) ── → ~/.m2/repository
  │
  └── deploy            (phase 23) ── → remote repository
```

### Running Default Lifecycle Phases

```bash
mvn validate             # stop after validate
mvn compile              # stop after compile (phases 1-7)
mvn test                 # stop after test (phases 1-15)
mvn package              # stop after package (phases 1-17)
mvn verify               # stop after verify (phases 1-21)
mvn install              # stop after install (phases 1-22)
mvn deploy               # run all 23 phases
```

## Lifecycle 2 — Clean

The **clean lifecycle** deletes files produced by the previous build. It has three phases:

| # | Phase | What Happens Here |
|---|---|---|
| 1 | **pre-clean** | Executes any processes needed before the actual cleaning (rarely used) |
| 2 | **clean** | Deletes the `target/` directory (all compiled classes, JARs, reports, etc.) |
| 3 | **post-clean** | Executes processes after cleaning completes (rarely used) |

### Why Clean Matters

Without running `clean`, leftover files from a previous build can interfere. A common issue is stale `.class` files from deleted `.java` files still being included in the JAR. Always run `mvn clean install` (or `mvn clean package`) in CI pipelines.

```bash
mvn clean               # runs pre-clean → clean → post-clean
mvn clean install       # clean lifecycle, then default up to install
mvn clean package -T 4  # clean then package using 4 threads
```

## Lifecycle 3 — Site

The **site lifecycle** generates project documentation and can deploy it to a web server. It has four phases:

| # | Phase | What Happens Here |
|---|---|---|
| 1 | **pre-site** | Executes processes before site generation |
| 2 | **site** | Generates the project documentation website (Javadoc, test reports, etc.) |
| 3 | **post-site** | Executes processes after site generation |
| 4 | **site-deploy** | Deploys the generated site to a web server (via SCP, FTP, etc.) |

```bash
mvn site              # generate site into target/site/
mvn site:run          # generate and serve site on localhost:8080
mvn site-deploy       # generate and deploy to configured server
```

The site includes (when configured): Javadoc, Surefire test reports, JaCoCo coverage, Checkstyle results, FindBugs/SpotBugs results, and dependency analysis.

---

## How Phases and Lifecycles Interact

Running a phase from one lifecycle **does not** trigger phases from another lifecycle. They are independent sequences.

```bash
# This runs TWO separate lifecycles:
mvn clean install
#    ↑               ↑
#    clean lifecycle  default lifecycle up to install

# This is NOT the same as:
mvn install
# (which skips the clean lifecycle entirely)
```

### Plugin Goals and Lifecycle Binding

Each phase in the lifecycle can have zero or more plugin goals bound to it. When Maven executes a phase, it runs all goals bound to that phase (and all preceding phases).

Default bindings for `jar` packaging:

| Phase | Bound Goal |
|---|---|
| `process-resources` | `maven-resources-plugin:resources` |
| `compile` | `maven-compiler-plugin:compile` |
| `process-test-resources` | `maven-resources-plugin:testResources` |
| `test-compile` | `maven-compiler-plugin:testCompile` |
| `test` | `maven-surefire-plugin:test` |
| `package` | `maven-jar-plugin:jar` |
| `install` | `maven-install-plugin:install` |
| `deploy` | `maven-deploy-plugin:deploy` |

You can also bind your own plugin goals to any phase — see [Chapter 8 — Plugins](./08-plugins.md).
