## 4. Virtual Threads (Project Loom) — JEP 444 ⭐ GAME-CHANGING

### The Problem: Platform Threads Are Expensive

```
Traditional server model:
1 HTTP request → 1 platform thread
Platform thread ≈ 1 MB stack + OS scheduling overhead

// With 10,000 concurrent requests:
// 10,000 platform threads × 1 MB = 10 GB RAM just for threads
// OS scheduling overhead becomes the bottleneck
// This is why Node.js, async Python, Kotlin coroutines existed
```

### What Virtual Threads Are

Virtual threads are **lightweight threads managed by the JVM**, not the OS. They are **cheap to create** (millions possible), **blocked virtual threads don't consume OS threads**, and they are **transparent to existing Java code** — write blocking code that scales like async code.

```
Virtual thread blocked on I/O:
    JVM unmounts it from carrier (OS) thread
    Carrier thread serves another virtual thread
    When I/O completes, JVM remounts virtual thread
    Your code sees simple blocking — no async callbacks

Cost: ~few KB per virtual thread vs ~1 MB per platform thread
```

### Creating Virtual Threads

```java
// Method 1 — direct creation
Thread vt = Thread.ofVirtual().start(() -> {
    System.out.println("Running in virtual thread: " + Thread.currentThread());
    // Thread[#21,virtual] = <unnamed>,5,<unnamed>
});

// Method 2 — virtual thread per task executor (most common)
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10_000; i++) {
        executor.submit(() -> handleRequest());
    }
} // Auto-closes, waits for all tasks

// Method 3 — Thread.Builder
Thread.Builder builder = Thread.ofVirtual().name("worker-", 0);
Thread t1 = builder.start(task1);
Thread t2 = builder.start(task2);
```

### Real-World: HTTP Server with Virtual Threads

```java
// Spring Boot 3.2+ — enable virtual threads with one property:
// spring.threads.virtual.enabled=true
// That's it! Every @RestController request runs on a virtual thread

// Tomcat / Jetty with virtual threads — write blocking code that scales
@RestController
public class OrderController {

    // This blocking code now handles 100,000+ concurrent requests
    // No async, no reactive, no callbacks — plain blocking Java
    @GetMapping("/orders/{id}")
    public OrderDTO getOrder(@PathVariable Long id) {
        // These blocking calls don't waste OS threads
        Order order = orderRepo.findById(id).orElseThrow();     // Blocks → JVM parks VT
        Customer customer = customerService.get(order.getCustomerId()); // Blocks → parks
        Inventory inventory = inventoryService.check(order.getItems()); // Blocks → parks
        return mapper.toDTO(order, customer, inventory);
    }
}

// Manual virtual thread executor
@Service
public class BulkOrderProcessor {

    public List<OrderResult> processBulkOrders(List<OrderRequest> requests) {
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            List<Future<OrderResult>> futures = requests.stream()
                .map(req -> executor.submit(() -> processOrder(req)))
                .collect(Collectors.toList());

            return futures.stream()
                .map(f -> {
                    try { return f.get(); }
                    catch (Exception e) { return OrderResult.failed(e.getMessage()); }
                })
                .collect(Collectors.toList());
        }
    }

    // Process 10,000 orders concurrently — no problem with virtual threads
    public void syncAllOrders(List<String> orderIds) {
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            orderIds.forEach(id ->
                executor.submit(() -> {
                    Order order = externalApi.fetchOrder(id);    // Blocking HTTP
                    orderRepo.save(order);                        // Blocking DB
                    searchIndex.index(order);                     // Blocking ES
                })
            );
        } // Blocks until all complete
    }
}
```

### Virtual Threads vs Platform Threads vs Reactive

```java
// Platform thread pool (old approach — limited concurrency)
ExecutorService platform = Executors.newFixedThreadPool(200); // Max 200 concurrent

// Virtual threads (new approach — massive concurrency)
ExecutorService virtual = Executors.newVirtualThreadPerTaskExecutor(); // Millions concurrent

// Reactive (complex — avoid if virtual threads work for your use case)
Mono<OrderDTO> reactive = orderRepo.findById(id)   // Non-blocking
    .flatMap(order -> customerService.get(order.getCustomerId()))
    .flatMap(customer -> ...);
// Complex, hard to debug, different programming model

// With virtual threads — write the simple blocking version, get the performance
OrderDTO result = orderRepo.findById(id)             // Blocking
    .flatMap(order -> customerService.get(...));      // Not needed — just chain normally
```

### Corner Cases and Gotchas

**1. Pinned virtual threads — the most important gotcha**
```java
// Virtual threads are "pinned" to the carrier OS thread when:
// a) Inside a synchronized block
// b) Calling a native method

// PROBLEM — synchronized pins the virtual thread, blocking the carrier thread
// This defeats the purpose of virtual threads in high-concurrency scenarios
public synchronized void badMethod() {
    dbCall(); // Pinned! Carrier thread blocked during this database call
}

// SOLUTION 1 — use ReentrantLock instead of synchronized
private final ReentrantLock lock = new ReentrantLock();

public void goodMethod() {
    lock.lock();
    try {
        dbCall(); // Virtual thread can be parked here — no pinning
    } finally {
        lock.unlock();
    }
}

// SOLUTION 2 — minimise scope of synchronized
public void betterMethod() {
    prepareData(); // Outside sync — virtual thread can park here
    synchronized(this) {
        updateSharedState(); // Keep sync blocks very short and CPU-only
    }
    persistData(); // Outside sync — can park here too
}

// Check for pinning during development:
// java -Djdk.tracePinnedThreads=full MyApp
```

**2. Thread locals — be careful with thread-per-request assumptions**
```java
// ThreadLocal still works with virtual threads BUT:
// With millions of virtual threads, ThreadLocal memory can be significant
// Java 21 introduces ScopedValue as the better alternative (see below)

// CAUTION: thread pools that reuse threads — virtual threads are NOT reused
// Each virtual thread executes ONE task, then is discarded
// ThreadLocal data from previous task is NOT present — good for isolation
// BUT if you WANT to carry state across tasks in a pool, use ScopedValue

// Spring's SecurityContextHolder (ThreadLocal-based) works fine with virtual threads
// because Spring Boot correctly manages the security context per request
```

**3. Don't pool virtual threads — anti-pattern**
```java
// WRONG — don't pool virtual threads (defeats the purpose)
ExecutorService pool = Executors.newFixedThreadPool(1000); // Platform thread pool hack
// Using virtual threads as if they were platform threads

// RIGHT — use one virtual thread per task
ExecutorService vt = Executors.newVirtualThreadPerTaskExecutor();
// The JVM manages the actual OS thread pool internally (fork-join pool)
```

**4. CPU-bound tasks — virtual threads don't help**
```java
// Virtual threads excel at I/O-bound tasks (database, HTTP, file, sleep)
// For CPU-bound tasks, use platform thread pools matching CPU core count

// CPU-bound (image processing, cryptography, ML inference)
ExecutorService cpuBound = Executors.newFixedThreadPool(
    Runtime.getRuntime().availableProcessors()
);

// I/O-bound (database, REST calls, message queue)
ExecutorService ioBound = Executors.newVirtualThreadPerTaskExecutor();
```

## 5. Pattern Matching for Switch (Final) — JEP 441

Combines `switch` with type patterns — the most expressive Java feature since Streams.

```java
// Basic pattern matching switch
Object obj = getValue();

String result = switch (obj) {
    case Integer i  -> "Integer: " + i;
    case Long l     -> "Long: " + l;
    case Double d   -> "Double: " + d;
    case String s   -> "String: " + s;
    case null       -> "null";
    default         -> "Unknown: " + obj.getClass().getSimpleName();
};
```

### Guarded Patterns (when clauses)

```java
// 'when' clause adds conditions to a pattern
String classify(Number n) {
    return switch (n) {
        case Integer i when i < 0    -> "Negative integer: " + i;
        case Integer i when i == 0   -> "Zero";
        case Integer i when i > 1000 -> "Large integer: " + i;
        case Integer i               -> "Integer: " + i;
        case Double d when d.isNaN() -> "Not a number";
        case Double d                -> "Double: " + d;
        default                      -> "Other number";
    };
}
```

### Real-World: JSON/Object Graph Processing

```java
// Processing a heterogeneous message bus
public class MessageRouter {

    public Response route(Message message) {
        return switch (message) {

            // Type + field extraction + guard
            case CreateOrderMessage m when m.items().isEmpty() ->
                Response.badRequest("Order must have items");

            case CreateOrderMessage m when m.totalAmount().compareTo(LIMIT) > 0 ->
                Response.badRequest("Order exceeds limit");

            case CreateOrderMessage m ->
                orderService.create(m.customerId(), m.items(), m.totalAmount());

            case CancelOrderMessage m ->
                orderService.cancel(m.orderId(), m.reason());

            case PaymentMessage m when m.isRefund() ->
                paymentService.processRefund(m.orderId(), m.amount());

            case PaymentMessage m ->
                paymentService.processPayment(m.orderId(), m.amount());

            case null ->
                Response.badRequest("Null message received");

            default ->
                Response.badRequest("Unknown message type: " + message.getClass());
        };
    }
}

// Processing a polymorphic API response
public BigDecimal extractPrice(Object apiResponse) {
    return switch (apiResponse) {
        case Map<?, ?> m when m.containsKey("price") ->
            new BigDecimal(m.get("price").toString());

        case Map<?, ?> m when m.containsKey("amount") && m.containsKey("currency") ->
            convertToCurrency(m.get("amount").toString(), m.get("currency").toString());

        case String s when s.matches("\\d+\\.?\\d*") ->
            new BigDecimal(s);

        case Number n ->
            new BigDecimal(n.toString());

        case null ->
            BigDecimal.ZERO;

        default ->
            throw new IllegalArgumentException("Cannot extract price from: " + apiResponse);
    };
}
```

### Sealed Classes + Pattern Matching = Perfect Combination

```java
// With sealed hierarchy, switch is exhaustive — no default needed
public sealed interface Notification
    permits EmailNotification, SMSNotification, PushNotification, WebhookNotification {}

public record EmailNotification(String to, String subject, String body) implements Notification {}
public record SMSNotification(String phoneNumber, String message) implements Notification {}
public record PushNotification(String deviceToken, String title, String payload) implements Notification {}
public record WebhookNotification(String url, String payload, Map<String,String> headers) implements Notification {}

public class NotificationDispatcher {

    public NotificationResult dispatch(Notification notification) {
        return switch (notification) {
            case EmailNotification e ->
                emailClient.send(e.to(), e.subject(), e.body());

            case SMSNotification s when s.message().length() > 160 ->
                smsSplitter.sendLong(s.phoneNumber(), s.message());

            case SMSNotification s ->
                smsClient.send(s.phoneNumber(), s.message());

            case PushNotification p ->
                pushClient.send(p.deviceToken(), p.title(), p.payload());

            case WebhookNotification w ->
                httpClient.post(w.url(), w.payload(), w.headers());
            // No default needed — sealed type, all cases covered
        };
    }
}
```

## 6. Record Patterns (Final) — JEP 440

Deconstruct record components directly inside `instanceof` and `switch` patterns.

```java
// Basic record pattern
record Point(int x, int y) {}
record Line(Point start, Point end) {}

Object obj = new Point(3, 4);

// Old way — extract manually
if (obj instanceof Point p) {
    int x = p.x(); int y = p.y();
    System.out.println(x + ", " + y);
}

// Record pattern — destructure in one step
if (obj instanceof Point(int x, int y)) {
    System.out.println(x + ", " + y); // x and y directly in scope
}

// Nested record patterns
if (obj instanceof Line(Point(int x1, int y1), Point(int x2, int y2))) {
    double length = Math.sqrt(Math.pow(x2-x1, 2) + Math.pow(y2-y1, 2));
}
```

### Real-World: Domain Object Deconstruction

```java
// Domain records
record Money(BigDecimal amount, String currency) {}
record OrderItem(String productId, int quantity, Money price) {}
record Order(String id, List<OrderItem> items, Money total, OrderStatus status) {}

// Record patterns in switch
public String formatOrderSummary(Object obj) {
    return switch (obj) {
        case Order(var id, var items, Money(var amount, var currency), var status) ->
            "Order %s: %s items, %s %s (%s)".formatted(
                id, items.size(), amount, currency, status);

        case OrderItem(var productId, var qty, Money(var price, var currency)) ->
            "%s × %d @ %s %s".formatted(productId, qty, price, currency);

        case Money(var amount, var currency) ->
            amount + " " + currency;

        default -> obj.toString();
    };
}

// Practical: processing order events with nested records
public void processEvent(OrderEvent event) {
    switch (event) {
        case OrderPlaced(var orderId, var customerId,
                         Money(var amount, var currency),
                         var occurredAt) -> {
            log.info("Order {} placed by {} for {} {}", orderId, customerId, amount, currency);
            notificationService.sendConfirmation(customerId, orderId);
            analyticsService.trackPurchase(customerId, amount, currency);
        }

        case OrderCancelled(var orderId, var reason, var occurredAt) -> {
            log.info("Order {} cancelled: {}", orderId, reason);
            refundService.initiateRefund(orderId);
        }

        case OrderShipped(var orderId, var tracking, var occurredAt) ->
            notificationService.sendShippingUpdate(orderId, tracking);
    }
}
```

## 7. Sequenced Collections — JEP 431

A new set of interfaces that unify **ordered collection operations** across `List`, `Deque`, `LinkedHashSet`, `SortedSet`, `LinkedHashMap`, and `SortedMap`.

### The Problem They Solve

```java
// Before Java 21 — getting first/last element was inconsistent:
List<String> list = new ArrayList<>(List.of("a", "b", "c"));
String first = list.get(0);              // List
String last  = list.get(list.size()-1);  // List — verbose

Deque<String> deque = new ArrayDeque<>(List.of("a","b","c"));
String firstD = deque.getFirst();        // Deque
String lastD  = deque.getLast();         // Deque

LinkedHashSet<String> lhs = new LinkedHashSet<>(Set.of("a","b","c"));
String firstS = lhs.iterator().next();   // LinkedHashSet — ugly!
// Getting last element of LinkedHashSet was painful

// Java 21 — unified interface: SequencedCollection
// getFirst(), getLast(), reversed() on List, Deque, LinkedHashSet, SortedSet
```

### New Interfaces

```java
// SequencedCollection<E>
interface SequencedCollection<E> extends Collection<E> {
    SequencedCollection<E> reversed();
    void addFirst(E e);
    void addLast(E e);
    E getFirst();       // NoSuchElementException if empty
    E getLast();
    E removeFirst();
    E removeLast();
}

// SequencedSet<E> extends SequencedCollection + Set
// SequencedMap<K,V> extends Map
// - firstEntry(), lastEntry(), sequencedEntrySet(), sequencedKeySet(), sequencedValues()
```

### Real-World Examples

```java
public class OrderHistoryService {

    // Consistent first/last access across collection types
    public Optional<Order> getMostRecentOrder(SequencedCollection<Order> orders) {
        if (orders.isEmpty()) return Optional.empty();
        return Optional.of(orders.getLast()); // Works on List, Deque, LinkedHashSet
    }

    public Optional<Order> getOldestOrder(SequencedCollection<Order> orders) {
        if (orders.isEmpty()) return Optional.empty();
        return Optional.of(orders.getFirst());
    }

    // Reversed view — no copy, memory efficient
    public List<Order> getOrdersNewestFirst(List<Order> orders) {
        return orders.reversed(); // Returns a reversed VIEW — not a copy!
    }

    // LinkedHashMap now has ordered entry access
    public Map.Entry<String, BigDecimal> getMostExpensiveCategory(
            LinkedHashMap<String, BigDecimal> revenueByCategory) {
        return revenueByCategory.reversed().firstEntry();
        // Instead of: new ArrayList<>(map.entrySet()).get(map.size()-1)
    }

    // Audit trail — add to front
    public void recordAction(String action, Deque<String> auditLog) {
        auditLog.addFirst(action + " at " + Instant.now());
        if (auditLog.size() > 100) auditLog.removeLast(); // Keep last 100
    }
}
```

### Corner Cases

```java
// 1. reversed() returns a VIEW — changes to original are reflected
List<String> original = new ArrayList<>(List.of("a", "b", "c"));
List<String> reversed = original.reversed(); // View, not copy
original.add("d");
System.out.println(reversed); // [d, c, b, a] — reflected immediately

// 2. getFirst()/getLast() throw NoSuchElementException on empty
List<String> empty = new ArrayList<>();
empty.getFirst(); // THROWS NoSuchElementException
// Always check isEmpty() or use Optional pattern

// 3. SequencedSet — Set.of() and unmodifiable sets
Set<String> set = Set.of("a","b","c"); // NOT a SequencedSet — unordered
LinkedHashSet<String> lhs = new LinkedHashSet<>(List.of("a","b","c")); // IS a SequencedSet

// 4. reversed() on SequencedMap
LinkedHashMap<String, Integer> map = new LinkedHashMap<>();
map.put("a", 1); map.put("b", 2); map.put("c", 3);
map.reversed().forEach((k,v) -> System.out.println(k + "=" + v));
// c=3, b=2, a=1
```

## 8. Structured Concurrency (Preview) — JEP 453

Structured Concurrency treats a group of tasks as a single unit — ensuring that if any task fails, all others are cancelled, and the code structure matches the task lifecycle.

```java
// Problem with CompletableFuture — tasks can leak if one fails
// No automatic cancellation of sibling tasks

// Java 21 Structured Concurrency — tasks are structured like try blocks
public ProductPage getProductPage(Long productId) throws Exception {

    try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
        // Fork tasks — all run concurrently
        StructuredTaskScope.Subtask<Product> productTask =
            scope.fork(() -> productRepo.findById(productId));

        StructuredTaskScope.Subtask<List<Review>> reviewsTask =
            scope.fork(() -> reviewService.findByProduct(productId));

        StructuredTaskScope.Subtask<InventoryStatus> inventoryTask =
            scope.fork(() -> inventoryService.getStatus(productId));

        // Wait for ALL tasks OR first failure
        scope.join().throwIfFailed();
        // If ANY task throws, all others are cancelled — no resource leaks

        // All succeeded — safely access results
        return new ProductPage(
            productTask.get(),
            reviewsTask.get(),
            inventoryTask.get()
        );
    }
}

// ShutdownOnSuccess — return first successful result (like anyOf but cleaner)
public String findProductById(String id) throws Exception {
    try (var scope = new StructuredTaskScope.ShutdownOnSuccess<String>()) {
        scope.fork(() -> primaryDatabase.findProduct(id));
        scope.fork(() -> fallbackDatabase.findProduct(id));
        scope.fork(() -> cacheService.findProduct(id));

        scope.join(); // Returns when FIRST succeeds
        return scope.result(); // Gets the first successful result
    }
}
```

## 9. Scoped Values (Preview) — JEP 446

A better alternative to `ThreadLocal` for virtual threads — immutable, inheritable, and efficient.

```java
// ThreadLocal problems with virtual threads:
// - Mutable (can be changed accidentally by any code)
// - Inherited but not automatically cleaned up
// - Memory overhead with millions of virtual threads

// ScopedValue — immutable within a scope, inherited by child threads
public class SecurityContext {

    // Declare scoped value — like a typed "channel" for data
    static final ScopedValue<User> CURRENT_USER = ScopedValue.newInstance();
    static final ScopedValue<String> CORRELATION_ID = ScopedValue.newInstance();

    // Run with a bound value — replaces ThreadLocal.set()
    public Response handleRequest(Request request) {
        User user = authenticate(request);
        String correlationId = request.getHeader("X-Correlation-ID");

        return ScopedValue
            .where(CURRENT_USER, user)
            .where(CORRELATION_ID, correlationId)
            .call(() -> processRequest(request));
        // Values are automatically unbound when the scope exits
    }

    // Deep in the call stack — access scoped value
    public void auditAction(String action) {
        User user = CURRENT_USER.get();         // Always available in this scope
        String corrId = CORRELATION_ID.get();
        auditLog.record(user.id(), action, corrId);
    }
}

// Real-world: multi-tenant system with virtual threads
public class TenantService {

    static final ScopedValue<Tenant> CURRENT_TENANT = ScopedValue.newInstance();

    public void processForTenant(Tenant tenant, List<Request> requests) {
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            requests.forEach(req ->
                executor.submit(() ->
                    ScopedValue.where(CURRENT_TENANT, tenant)
                               .run(() -> processRequest(req))
                    // Each virtual thread has tenant in scope
                    // No risk of leaking tenant context to wrong thread
                )
            );
        }
    }

    // Deep in call stack — tenancy is available without passing it through params
    private void validateBusinessRule(Rule rule) {
        Tenant tenant = CURRENT_TENANT.get();
        if (!tenant.isRuleApplicable(rule)) {
            throw new TenantRuleException(rule, tenant);
        }
    }
}
```

## 12. Performance Improvements in Java 21

### Generational ZGC (JEP 439)

```bash
# Enable Generational ZGC — collects young objects more frequently
# Lower pause times for most workloads
java -XX:+UseZGC -XX:+ZGenerational MyApp

# G1 remains the default GC
# ZGC is best for: large heaps, latency-sensitive, microsecond pause times
```

### Comparison of Modern Java GC Options

| GC | Best For | Pause Times | Throughput |
|----|---------|-------------|-----------|
| G1 (default) | General purpose | < 200ms | High |
| ZGC | Latency-sensitive | < 1ms | Good |
| Generational ZGC (21+) | Latency + throughput | < 1ms | Better |
| Shenandoah | Low-latency | < 10ms | Good |
| Serial | Single-threaded / tiny heap | - | - |
